# Page Tabs Design

Goal: Redefine every visible app tab by purpose, remove the previous one-size layout rule from non-operational pages, and keep Orders/Routes feature-complete for v1 without adding new behavior.

## Scope
- Keep `/app/orders`, `/app/routes`, and `/app/routes/:routeId` functionally unchanged.
- Replace placeholder Analytics, Workflows, Drivers & Vehicles, Settings, and Additional screens with concise purpose-led page structures.
- Settings must explicitly separate user-controlled variables from runtime/system values generated by Shopify orders, delivery routes, and analytics.

## Page roles
- Orders: operational intake and route-plan creation from Shopify orders. No new features in this pass.
- Routes: persisted route plan list and route detail review. No new features in this pass.
- Analytics: explain the operational signals the app will derive from orders and route plans, without introducing fake controls or metrics.
- Workflows: describe the delivery lifecycle and automation boundaries, without enabling new workflow actions.
- Drivers & Vehicles: define the resource-management surface for named drivers, vehicles, assignments, and readiness, without adding persistence.
- Settings: own merchant-editable variables such as departure location and future planning defaults; distinguish them from read-only runtime facts.
- Additional: non-nav parking lot for future modules, kept low-prominence and purpose-labeled.

## UI approach
Create a flexible page shell for non-Orders/Routes pages: header, optional eyebrow, sections, status pills, value lists, and compact grids. Use plain React and existing styling conventions. Do not add dependencies.

## Validation
Source tests should prove:
- old layout enforcement no longer applies to all standard tabs;
- non-Orders/Routes tabs use role-specific copy and `PageShell` sections;
- Settings names user variables and runtime/system values separately;
- Orders and Routes test coverage remains intact and no new feature controls are added there.
